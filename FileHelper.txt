  设计 · 第 2 段：数据模型 + Reconciler 机制

  这是解决你核心诉求（Explorer 移动不丢标签）的关键部分，细一点。

  数据模型（SQLite 表）

  FileRecord                          Tag
  ─────────────                       ─────────────
  id          UUID (主键)             id          UUID
  content_hash  SHA-256  ◄── 查找键   name        TEXT UNIQUE
  current_path  TEXT                   color       TEXT (可选, UI 用)
  size        INTEGER                 usage_count INTEGER (冗余, 排序用)
  mtime       INTEGER                 created_at  TIMESTAMP
  status      active|missing|deleted
  first_seen  Timestamp               FileTag (多对多)
  last_seen   Timestamp               ─────────────
                                    file_id     FK
                                    tag_id      FK
                                    source      manual|rule|ai
                                    confidence  REAL (0-1, AI 用)
                                    added_at    Timestamp

  关键设计：
  - UUID 是文件身份，hash 只是查找键。两个相同内容的
  PDF（哈希一样）各自有独立标签 —— 不会被合并
  - status 三态：active（在）/ missing（扫描时没找到，给 N 天宽限期）/
  deleted（彻底删除，但标签留 30 天可恢复）

  ---
  Reconciler 算法（每次 scan 必跑）

  对每个被扫到的文件 F (路径 P, 哈希 H):
    ┌─ 路径匹配：找 current_path == P 的记录 R_path
    ┌─ 哈希匹配：找 content_hash == H 的记录 R_hash

    ① R_path == R_hash (或都为空且新建)
         → 命中现有记录，更新 last_seen ✓

    ② R_hash 存在, R_hash.current_path != P
         → MOVE！用户在 Explorer 里移动了文件
         → 更新 R_hash.current_path = P，标签全保留 ✓✓

    ③ R_path 存在, R_path.content_hash != H
         → EDIT！同路径但内容变了
         → 创建新 FileRecord，提示用户:
           "检测到 [文件名] 被修改，是否迁移旧标签？[Y/n]"

    ④ 都不存在
         → NEW，创建记录，无标签

  scan 结束后:
    所有 last_seen < 本次 scan 开始时间的 active 记录 → missing
    missing 超过 N 天 → deleted (但 FileTag 行保留 30 天)

  为什么这套能 work：
  - 用户在 Explorer 把 C:\A\x.pdf 拖到 D:\B\x.pdf —— 哈希不变，下次扫描命中
  ②，标签全保留
  - 用户改了 PDF 内容 —— 命中 ③，工具主动问要不要迁移标签
  - 用户彻底删了文件 —— 进 missing，再进 deleted，30 天内还能通过 sidecar 恢复

  ---
  Sidecar 格式（备份/同步用，非主存储）

  {
    "schema_version": 1,
    "content_hash": "sha256:abc...",
    "original_path": "C:/Documents/paper.pdf",
    "tags": [{"name": "ai/ml", "source": "manual"},
             {"name": "to-read", "source": "ai", "confidence": 0.92}],
    "exported_at": "2026-06-19T17:30:00"
  }

  用途： filetidy export 批量导出 → 上 git → 另一台机器 filetidy import
  导入。不作为运行时存储，避免前文提到的 sidecar 被孤儿的问题。

  ---
  这段设计的几个关键点你重点看一下：

  1. UUID 是身份、hash 是查找键 ——
  这套让你能区分"两个内容相同的文件各有各的标签"
  2. Reconciler 的 4 种情况（命中/MOVE/EDIT/NEW）—— 看看 EDIT
  的处理你认不认可（主动问要不要迁移标签，而不是默认迁移或默认不迁移）
